zm-next — CUDA pipeline & LLM event review

Capability summary, weighted toward the zero-copy decode/detect path.

zm-next is a plugin-based video-pipeline engine (next-gen ZoneMinder core). C++20, CMake + vcpkg. A zm-core process loads a JSON pipeline, dlopens plugins (capture → decode → detect → output), runs each stage in its own thread, and passes frames over a stable C ABI (zm_frame_hdr_t).

Reference box for the numbers below: RTX 5070 Ti (16 GB, sm_120 Blackwell), Ryzen 7 9700X. Stack: FFmpeg 8 (nvcodec/x264/x265), ONNX Runtime GPU (cuda12), CUDA 12.9 / cuDNN 9, vcpkg x64-linux-dynamic.

1. Zero-copy decode → detect path

The decoded surface never leaves VRAM from decode through inference — no host readback of the image, no re-upload. Only a kilobyte-sized motion grid crosses PCIe.

Stages (all on-device)

  1. NVDEC decode → NV12 surface in VRAM. FFmpeg hw_frames_ctx yields a CUDA device pointer (Y plane + interleaved UV), never a host buffer.
  2. On-GPU motion gate. A CUDA kernel downsamples luma to a small grid (launch_luma_grid) and diffs it against the previous frame's grid. Only the grid crosses PCIe (a few KB), never the full frame. Optional whole-scene luma-jump suppression (headlights / IR-cut / auto-gain) is available as a config knob.
  3. Preprocess NV12 → CHW on-device. A fused kernel (launch_nv12_to_chw, detect_cuda.cu) samples the NV12 surface — optionally just a motion ROI crop — into a planar normalized 3×net×net float tensor: bilinear letterbox scale/pad, BT.601 YUV→RGB, /255, matching the CPU path.
  4. Inference via ORT CUDA EP + IoBinding. The session input is bound directly to the device tensor from step 3. NMS-free output is decoded to source-pixel boxes on-device, mapped back through the letterbox.

Per-call scratch buffers are thread_local. The CUDA context uses cudaDeviceScheduleBlockingSync (blocking, not spin-wait), keeping detect-thread CPU low.

Measured (1080p h264, single stream, yolo26n)

Detect's VRAM footprint is ~720 MB (model + ORT/cuDNN + NVDEC surfaces).

Shared inference engine (multi-camera)

A single shared batched InferenceEngine serves all camera threads: each thread preprocesses zero-copy and hands its device tensor to one ORT session that batches them. At 8 cameras this holds CPU around 30% and uses roughly half the VRAM of per-camera sessions, at the same throughput. The session is a process-lifetime singleton.

Fused decode+detect stage

The NVDEC surface is call-scoped — valid only inside on_frame. The decode_detect plugin composes decode internally (via dlopen) and runs detect synchronously in the same on_frame, so the surface lives across both stages with no copy. Surface lifetime is managed with av_frame_clone / av_frame_free.

Portability — HwBackend

HwBackend abstracts the device path as acquire / release / motion / preprocess / infer. CUDA is the implemented backend; the interface leaves clean seams for VideoToolbox/Metal (Apple), OpenVINO (Intel), ROCm (AMD), DirectML (Windows) without touching plugin logic. The frame ABI carries hw_type (CPU/CUDA/VAAPI/VideoToolbox/DXVA).

2. LLM event review

llm_event_review is a DETECT-triggered plugin: on a detection/alert event it grabs the latest decoded frame for that stream, JPEG-encodes it (downscaled so longest-edge area ≤ ~1 MP), and asks a vision-LLM to describe the event.

  • Model-agnostic. Speaks the OpenAI-compatible /v1/chat/completions vision API, so the model is a config string + server URL — Qwen3-VL / Qwen2.5-VL / MiniCPM-V / InternVL, local or cloud, with no rebuild. IVisionProvider interface.
  • Per-track debounce on wall-clock (steady_clock, pruned) so a tracked object isn't reviewed repeatedly.

Local serving (vLLM on the 5070 Ti)

vLLM 0.23, torch 2.11+cu130. Default model Qwen/Qwen3-VL-4B-Instruct-FP8 (8B also supported).

  • On 16 GB: --enforce-eager + a capped --max-model-len are required (otherwise KV cache can't fit and engine init fails).
  • 4B @ gpu-mem-util 0.60 / max-model-len 4096 → ~9.5 GB used, ~6 GB free for the detector.
  • Warm latency ~0.5–0.8 s/event (single frame); ~1.5 s for a 6-frame montage.

Sharing one GPU with the detector

  • VRAM: detect needs ~720 MB; with the 4B at ~9.5 GB there is comfortable headroom.
  • Compute: the SMs time-slice. Detect runs at 1.80 ms idle and ~15 ms while the LLM is actively generating — within budget for a few 30 fps cameras (33 ms/frame); review is low-QPS and debounced.

3. Integrated pipeline + metadata

End-to-end: decode_ffmpeg → detect_onnx → tracker → llm_event_review.

  • Tracker (tracker_core.hpp, SORT/IoU, persistent track_ids) provides stable counts. min_hits requires N IoU matches before a track is confirmed, so transient false detections are not counted; max_age keeps real tracks alive through detector dropouts.
  • VLM grounding. The VLM is grounded only on objective signals it cannot perceive from a still — trajectory/direction (from box-area trend), time, zone — and is not given the detector's count or confidence. Confidence is used as a router: high-confidence detections are trusted, marginal ones are blind-verified by the VLM (asked independently whether a subject is present), low ones dropped.
  • Motion via montage. Six stills tiled into one ~1 MP image let the model read direction, speed and loitering while staying ~1 image / ~1000 tokens.
  • WebVTT metadata tracks. Detections, tracker IDs+trails, and LLM captions each export as a WebVTT track (kind="metadata" for JSON boxes, kind="subtitles" for text), rendered as three independently-toggleable layers over the video for event playback.

TL;DR

  • Decode→detect stays in VRAM the whole way (NVDEC NV12 → on-GPU motion → on-GPU NV12→CHW → ORT CUDA EP via IoBinding). Only a KB-sized motion grid crosses PCIe. 387 fps end-to-end vs 31 fps all-CPU, detect 1.80 ms/inf, ~720 MB.
  • Fused stage keeps the call-scoped NVDEC surface alive across decode+detect; a shared engine serves N cameras from one session; HwBackend leaves portability seams.
  • LLM review is model-agnostic over an OpenAI-compatible API, runs Qwen3-VL-4B-FP8 locally in ~9.5 GB leaving room to co-run the detector, ~0.5–0.8 s/event, integrated behind the tracker.
