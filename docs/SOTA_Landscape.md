# zm-next / ZoneMinder — SOTA Research Landscape

_Generated 2026-06-26 from a multi-agent literature + ecosystem sweep (9 domains), synthesised and mapped to specific zm-next/zm-api components. Treat model names, versions, arXiv IDs and dates as **as-of mid-2026 and verify before adopting** — this space moves monthly. `recommendation` is adopt-now / pilot / watch / skip with (effort/impact)._

> Scope: zm-next (C++ plugin pipeline), zm-api (Rust control plane), and the apps (zm-dashboard, zm-mobile, zmNinjaNg). Competitive frame: Frigate, Scrypted, Viseron, Home Assistant.

## Executive summary

zm-next is genuinely near-SOTA on the hard parts almost no NVR gets right — the emit-ingredients plugin ABI, a shared batched ONNX engine with cross-camera dynamic batching, a multi-EP HwBackend with a validated fp16 ANE path, an NMS-free YOLO26 decode path already in code, and a local VLM describer that is architecturally ahead of Frigate GenAI. The wins are mostly model swaps and small wiring jobs on plumbing you already own, NOT rewrites. The four biggest gaps are all execution, not capability: (1) no shipped semantic/NL event search (the single largest competitive gap vs Frigate 0.15-0.17, and your own sqlite-vec design is correct but unbuilt); (2) the tracker is 2017-era SORT (no Kalman, ReID hooks present but fed a colour histogram); (3) privacy_mask is static-polygon only when the market sells subject-tracking redaction; (4) describe_vlm throws away Qwen3-VL's temporal grounding by sending one frame to a non-batching server. Ship YOLO26+INT8, multi-frame VLM, OC-SORT+real ReID, EdgeTAM dynamic redaction, and the sqlite-vec search loop, and zm-next moves from at-parity to ahead of Frigate on detection, tracking, redaction, and local GenAI.

## Where we're already near / ahead of SOTA

- Plugin ABI / emit-ingredients architecture: dlopen C-ABI plugins, per-stage StageRunner threading, EventBus pub/sub — a cleaner detector/stage abstraction than Frigate's process-per-detector (which forbids mixing backends) and the natural home for cheap late-fusion of audio+pose+detect signals.
- detect_onnx shared batched InferenceEngine (detect_engine.cpp/.hpp): ONE ORT session, cross-camera requests coalesced into one batched Run with a maxWaitUs linger knob and maxBatch cap — this is the DeepStream/Triton dynamic-batching pattern, the #1 many-camera efficiency lever, and Frigate does NOT have it.
- Multi-EP HwBackend (hw_backend_{cuda,metal,openvino,vaapi,vulkan}): EP-as-config-switch, with a bit-exact-validated fp16 Apple ANE path (~2ms, 8.4x vs CPU on M4 Pro; fp32 correctly noted as falling back to ~CPU). Ahead of every open NVR on the Apple story; the metal backend already references yolo26n.
- NMS-free decode already in code: detect_postprocess.hpp decode_nms_free() consumes the YOLO26 [num x 6] end-to-end output — the 2025 frontier decode path is plumbed before the model swap.
- describe_vlm + llm_event_review: a real LOCAL VLM (mlx-vlm Qwen3-VL) with a pluggable IVisionProvider abstraction and a track/alert-gated trigger — a strictly better VLM trigger than Frigate's thumbnail captioning or ha-llmvision's binary-sensor, and llm_event_review already emits structured {description, threat_level, confidence} reasoning events.
- Modular detect tier breadth: detect_onnx/seg/openvocab/pose + tracker + recognize_face + lpr + audio_detect + analytics_rules matches or exceeds Frigate 0.17 / Scrypted / Viseron on the model layer.
- tracker appearance/ReID plumbing is end-to-end (Det.embedding, EMA track embedding, cosine_sim gate, appearance_threshold/weight) — the consumer exists; only a discriminative embedding producer is missing.
- review_export already exports soft per-object alpha mattes (downscaled sigmoid masks, feathered premultiply) — anticipating the synopsis/redaction quality axis most NVRs ignore; conceptually ahead of Frigate's preview-grid review and aligned with the BriefCam/Milestone synopsis direction.
- NL-search DESIGN (docs/Research_Motion_and_LLM_Review.md): SQL-routed counting, code-side relative-date resolution, in-DB sqlite-vec pre-filter, strict-citation grounding with event_id verify — these are exactly the agentic-RAG lessons the field keeps relearning, and ahead of Frigate's similarity-only search (design is near-SOTA; it just isn't built yet).

- **Ship YOLO26 (NMS-free) as the default detector + make INT8 the default export path** — Swap the default closed-set weights in detect_onnx to YOLO26 (native NMS-free, DFL-removed) and add a static INT8 PTQ export/calibration path. decode_nms_free() already matches its [num x 6] output and the CoreML fp16 ANE path is validated. INT8 holds near-FP32 mAP on YOLO26 (where v11/v12 drop 3-7%) for 1.5-3.3x CPU/OpenVINO/NPU throughput.
  - _Why now:_ Pure model-file swap on plumbing you already built — lowest effort, highest leverage. NMS-free = deterministic latency and clean ANE export; INT8 nearly doubles many-camera CPU/NPU throughput. Frigate just moved its Coral default to YOLOv9; staying on older YOLO is a quiet accuracy/throughput deficit.
- **Build the sqlite-vec NL/semantic event-search loop (embed-at-ingest + hybrid retrieval + rerank)** — Implement the documented-but-unbuilt search: embed llm_event_review/describe_vlm descriptions AND a multimodal image embedding (SigLIP2 / JinaCLIP v2) of the event/track-crop thumbnail into sqlite-vec (vec0) + FTS5, fuse with RRF (k=60), add a tiny cross-encoder reranker (bge-reranker-v2-m3 / Qwen3-Reranker-0.6B), then the SQL-routed grounded-answer step from the design doc. Use a Matryoshka embed model truncated to 256-384 dims.
  - _Why now:_ Single largest competitive gap vs Frigate 0.15-0.17. All inputs exist (crops, captions, tubes); nothing is indexed. The design is already near-SOTA and AHEAD of Frigate on grounded Q&A + counting — building it converts a backend lead into a visible product win. Cross-encoder rerank is the highest-ROI RAG lever (+17% Recall@5).
- **Upgrade the tracker to OC-SORT (Kalman + ORU/OCM/OCR) + ByteTrack two-stage association** — Add a per-track constant-velocity Kalman filter plus OC-SORT's observation-centric heuristics and a ByteTrack-style low-confidence second association pass to tracker_core.hpp Tracker::update. Pure C++/std, NO new dependency; port the algorithm from boxmot/motcpp rather than vendoring (license + weight). Fits the existing greedy loop and Track struct.
  - _Why now:_ Tracker is self-described '2017-era SORT, no Kalman' and matches purely on last-box IoU — it fragments on every occlusion gap, non-linear motion, and low detection cadence (the core surveillance failure modes). This flips us from TRAILING Frigate's association (they have Kalman+CMC) to leading. 700+ FPS on CPU, no model needed.
- **Wire a real learned ReID embedding (OSNet-x0.25 / LITE) into the existing tracker hooks** — Run a lightweight ONNX ReID head (OSNet x0.25, ~2-4MB) or LITE-style in-detector features and emit a per-box embedding into the 'embedding' field the tracker already consumes — replacing the weak normalized colour histogram. Runs on the validated ANE/CUDA path. boxmot's ONNX ReID export is directly reusable.
  - _Why now:_ Highest leverage because the CONSUMER already exists and just drops detections lacking a discriminative embedding. Turns the cosmetic gate into SOTA-grade ID stability through occlusion and is the prerequisite for any cross-camera (MTMC) work — open competitive ground Frigate/Scrypted don't touch.
- **EdgeTAM dynamic, subject-tracking privacy redaction (the missing GDPR/DSAR feature)** — Add a 'dynamic' mode to privacy_mask (and a review_export/zm-api export path) that seeds EdgeTAM (Apache-2.0, distilled SAM2, 16 FPS on iPhone 15 Pro Max, real-time on M4 Pro/ANE) from the YOLO/detect_pose/recognize_face/lpr box you already produce, propagates a temporally-stable tight mask across the clip, and feathers+pixelates it. EfficientTAM as the Linux/CUDA variant.
  - _Why now:_ privacy_mask is static-polygon only (Black/Blur/Pixelate on fixed regions) — it cannot anonymise a person walking through frame, which is the actual GDPR/CCPA/DSAR ask competitors (Sighthound Redactor, Facit, Milestone) sell. zm-next's fully-local posture makes 'private by design' redaction-on-export a feature LEAD, not catch-up. EgoBlur (Apache-2.0 face+plate) is the simpler drop-in if EdgeTAM is too much initially.
- **Multi-frame VLM + vllm-mlx serving + structured JSON output for describe_vlm/llm_event_review** — Three coupled fixes: (1) send 4-8 track keyframes with per-frame timestamps instead of one JPEG (exploits Qwen3-VL interleaved-MRoPE temporal grounding); (2) replace mlx_vlm.server with vllm-mlx for continuous batching (3.7x) + content-hash multimodal prefix caching (cached image 21.7s->0.78s, ~28x); (3) enforce a minimal JSON schema via the llguidance mlx-vlm already bundles. Also upgrade the model from the MTP-hacked Qwen3.5-2B-OptiQ-4bit to Qwen3-VL-4B-Instruct-4bit.
  - _Why now:_ The plugin sends ONE frame, discarding exactly the temporal capability that makes Qwen3-VL worth running (loiter, package-drop, approach-then-leave). The default server has neither batching nor vision caching, so a multi-camera NVR re-encodes overlapping frames and serializes concurrent events. OpenAI-compatible wire means the existing curl client works unchanged.
- **New detect_action skeleton-sequence classifier consuming detect_pose** — Add a lightweight DETECT-stage plugin that windows detect_pose's per-person COCO-17 keypoints into a small TCN/LSTM/tiny-Transformer head and emits fall/fight/person-down/collapse 'action' events into analytics_rules. The keypoint stream (x/y/visibility, stream_id, pts_usec) is already the exact model input; 30-frame window -> head is the dominant 2024-2025 recipe (97-98% fall accuracy, real-time on CPU/ANE).
  - _Why now:_ detect_pose emits high-quality skeletons that NOTHING consumes temporally — analytics_rules works purely on bbox geometry, so loiter is just a dwell timer and fall/fight aren't recognized at all. The prosumer NVR field (Frigate/Scrypted/Viseron) ships NO skeleton-action recognition; this is a genuine differentiator, not catch-up.

## Quick wins (low effort / high impact)

- Swap default detector weights to YOLO26 — decode_nms_free() and the fp16 CoreML/ANE path already exist; near-zero code change.
- Swap the open-vocab models to re-exported YOLOE-26 (NMS-free, +10-11 AP over YOLO-World-L on LVIS, CoreML/ANE export) in detect_openvocab — same baked-prompt workflow, immediate accuracy/speed jump.
- Swap audio_detect's ONNX + labels to CED or EfficientAT (49.0 / 0.483 mAP vs YAMNet ~0.37) — model-agnostic plugin already supports it (add a log-mel front-end since these aren't raw-waveform).
- Upgrade describe_vlm model from the MTP-hacked Qwen3.5-2B-OptiQ-4bit to Qwen3-VL-4B-Instruct-4bit (already in mlx-vlm 0.6.3) — one model-id change in run_mlx_vlm_server.sh.
- Enforce a minimal JSON schema on VLM output via the llguidance mlx-vlm already bundles — removes brittle regex parsing, keeps the 0x0305 reasoning TLV clean.
- Add fp16-by-default export for the CoreML EP (fp32 silently falls back to ~CPU) and validate op coverage to avoid ANE<->CPU subgraph round-trips (e.g. reflect-pad).
- Add a cheap pose-feature fall heuristic (head-below-hip, low bbox aspect ratio, sustained immobility) to analytics_rules — catches falls with ZERO new model, gates detect_action later.
- Fix docs/MEMORY: describe_vlm runs on the Metal GPU, NOT the ANE (MLX/llama.cpp don't use ANE for transformer decode); the real ANE win is the CNN detect path. Mislabeling skews capacity planning.
- Replace review_matte's 3x3 box-blur feather with a fast guided-filter edge refine to snap coarse 160x160 YOLO-seg proto masks to true RGB edges — removes the blocky-cutout look in synopsis and any privacy feathering.

## Competitive gaps (features Frigate/Scrypted/Viseron ship that we don't)

- Semantic / natural-language event search: Frigate 0.15-0.17 ships CLIP-embedding semantic search, visual 'find more like this', and 'search triggers'. zm-next has the design (sqlite-vec, SQL router, strict-citation) but nothing built — the single biggest shipped-feature gap.
- GenAI review summaries surfaced in a timeline/review UI: Frigate 0.17 'Review' shows LLM title+description+activity-class (dangerous/suspicious/normal) inline. zm-next emits structured reasoning events (llm_event_review) but doesn't surface them in a review UX yet.
- Subject-tracking privacy redaction: competitors (Sighthound Redactor, Facit, Milestone/Brighter AI) sell automatic face/body/plate tracking+blur for GDPR/DSAR export; zm-next privacy_mask is static-polygon only.
- Tracking ASSOCIATION quality: Frigate has Kalman + optical-flow camera-motion compensation (customized Norfair); zm-next tracker has neither (greedy IoU, no Kalman, no CMC) — though zm-next leads on the detect/ReID/ANE stack Frigate lacks.
- Off-the-shelf edge-accelerator breadth: Frigate ships turnkey Coral/EdgeTPU, Hailo-8, RKNN, OpenVINO-NPU backends; zm-next has CUDA/CoreML/OpenVINO/VAAPI/Vulkan but no Coral/Hailo/RKNN — the cheap-edge many-camera path competitors lean on.
- Standardized live signaling: go2rtc/Frigate use WHEP/WHIP for ~0.5s WebRTC with standard players; zm-next uses a bespoke Node signaling bridge.
- H.265-over-WebRTC passthrough: output_webrtc is H.264-only (addH264Codec / H264RtpPacketizer), forcing transcode of camera-native H.265; Chrome 127+/Safari/iOS now support HEVC-over-WebRTC.
- Audio speech-to-text: Frigate 0.17 adds local transcription (sherpa-onnx/faster-whisper) feeding searchable speech events; zm-next has audio_detect (classification) but no STT.
- Audio tagging accuracy: audio_detect uses a YAMNet-style backbone (~0.37 mAP) where CED (49.0 mAP) / EfficientAT (0.483 mAP) are SOTA and a near-drop-in swap.

## Per-domain detail (with references)

### Object detection & open-vocabulary detection SOTA (2023-2025), mapped to zm-next detect_onnx / detect_openvocab

zm-next is closer to SOTA than expected. detect_onnx already has the right plumbing for the 2025 frontier: a YOLO26-style NMS-free decode path (decode_nms_free in detect_postprocess.hpp), letterbox preprocess, CoreML EP on Apple, a CUDA batched InferenceEngine, and a swappable HwBackend. The single biggest, lowest-effort win is a MODEL swap, not a code rewrite: move the default closed-set detector from your current YOLO (likely v8/v11-class) to YOLO26 (Sept 2025, Ultralytics) — it is natively NMS-free + DFL-free, exports cleanly to CoreML/ANE and ONNX, gives up to 43% faster CPU inference, better small-object accuracy (ProgLoss+STAL), and its end-to-end output already matches your decode_nms_free [num x 6] format. That is an adopt-now. The accuracy frontier has shifted from YOLO to real-time DETRs: RF-DETR (Roboflow, ICLR 2026; first real-time model >60 AP on COCO; Nano at 2.3ms/T4) and the D-FINE/DEIM family (54-56 AP) now beat YOLO11/12 at comparable latency, and both export to ONNX/TensorRT/CoreML — these are a strong PILOT for a high-accuracy tier on CUDA boxes, but their transformer decode head differs from YOLO so they need a new postprocess branch. On open-vocabulary, your detect_openvocab is one generation behind: it only consumes prompt-BAKED static ONNX. The SOTA is YOLOE (ICCV 2025, +3.5 AP over YOLO-Worldv2 at 1.4x speed, text/visual/prompt-free, re-parameterizes to zero-overhead YOLO at inference) and now YOLOE-26 (Feb 2026, NMS-free, CoreML/ANE export, 40.6 AP LVIS text-prompt). Adopt YOLOE/YOLOE-26 to replace whatever you bake today, and PILOT a small CLIP text-encoder ONNX to unlock true runtime-dynamic prompts (the 'FUTURE ENHANCEMENT' your own code comment flags). Heavy grounding models (Grounding DINO 1.5/1.6, T-Rex2, OWLv2) are accuracy leaders but are server-class/offline — WATCH or use only as an auto-labeling/distillation source, not in your real-time path. Net: you are near-SOTA on architecture/plumbing; the gaps are (1) ship YOLO26 as default, (2) add a DETR postprocess branch for the accuracy tier, (3) modernize open-vocab to YOLOE-26 + dynamic text prompts.

**Findings:**

- [**adopt-now** · low/high] **YOLO26 as the new default closed-set detector** → `detect_onnx (default model)`
  - SOTA: Ultralytics YOLO26 (arXiv 2509.25164, Sept 2025): native end-to-end NMS-free, DFL removed, ProgLoss+STAL for small objects, MuSGD optimizer. COCO 40.9-57.5 mAP at 1.7-11.8ms T4 TensorRT (n..x). Up to 43% faster CPU inference vs prior YOLO; INT8 retains ~FP32 mAP; exports to CoreML(ANE)/ONNX/TFLite/OpenVINO/TensorRT.
  - Relevance: Your decode_postprocess.hpp ALREADY has decode_nms_free() for the YOLO26 [num x 6] (x1,y1,x2,y2,conf,cls) end-to-end output, and detect_onnx already enables the CoreML EP. This is a model-file swap, not a code change. NMS-free = lower, more deterministic latency and simpler ANE export on your M4 Pro; 43% faster CPU helps Linux/no-GPU fallback.
- [pilot · medium/high] **RF-DETR as a high-accuracy real-time tier** → `detect_onnx (new high-accuracy model profile + DETR postprocess branch)`
  - SOTA: RF-DETR (Roboflow, ICLR 2026; github.com/roboflow/rf-detr): DINOv2 backbone, anchor-free, NMS-free. First real-time model >60 AP COCO (2XL=60.1 AP). RF-DETR-L 56.5 AP @6.8ms T4 FP16; RF-DETR-M 54.7 AP @4.52ms; RF-DETR-Nano 67.6 AP50 @2.3ms. Exports ONNX/TensorRT/CoreML. Apache-2.0.   - Relevance: Accuracy frontier has moved from YOLO to real-time DETR. BUT its transformer decode output differs from YOLO, so you need a new postprocess branch alongside decode_nms_free.
- [watch · medium/medium] **D-FINE / DEIM DETR family (alternative accuracy tier)** → `detect_onnx (DETR postprocess branch, candidate model)`
  - SOTA: D-FINE (ICLR 2025): redefines DETR regression as fine-grained distribution refinement; D-FINE-L 54.0 AP, D-FINE-X 55.8 AP COCO. DEIM (CVPR 2025, Dense O2O + MAL matching): DEIM-D-FINE-L/X 54.7/56.5 AP @124/78 FPS T4, ~50% less training time. Both beat YOLOv10-L/RT-DETR-R50 and YOLO11 across sizes.
  - Relevance: Same DETR postprocess branch as RF-DETR would cover these. D-FINE/DEIM are competitive accuracy/latency and are the academic SOTA references; worth benchmarking head-to-head with RF-DETR before committing the high-accuracy tier to one architecture. Lower priority than RF-DETR (less polished export/tooling).
- [watch · medium/medium] **RT-DETRv2/v3/v4 (mature DETR baseline)** → `detect_onnx (DETR postprocess validation model)`
  - SOTA: RT-DETRv2 (2407.17140, 2024): bag-of-freebies, discrete sampling for broad deployment. RT-DETRv3 (WACV 2025 oral): hierarchical dense positive supervision. RT-DETRv4 (2510.25257): leverages vision foundation models with no extra inference latency. Well-supported in HuggingFace Transformers + Ultralytics.
  - Relevance: RT-DETR is the most production-mature real-time DETR (broadest tooling, ONNX/TensorRT). If you want one DETR to validate the postprocess branch against before pursuing RF-DETR/D-FINE, RT-DETRv2 is the safe, well-documented choice. Slightly behind RF-DETR/D-FINE on accuracy now.

### Multi-object tracking, re-identification (ReID), and multi-target multi-camera (MTMC) tracking

zm-next's `tracker` plugin (plugins/tracker/tracker_core.hpp + tracker.cpp) is a greedy, highest-IoU one-to-one SORT-style associator with NO Kalman filter (self-described "v1, no Kalman"), an optional cosine-similarity appearance gate (off by default), class gating, and per-stream_id state driven off EventBus "detection" events. It is clean, dependency-free, and unit-tested, but algorithmically it sits at roughly 2017-era SORT — below the tracking-by-detection SOTA on every axis that matters for surveillance: occlusion robustness, non-linear motion, camera motion, and identity persistence through gaps. The single biggest, lowest-risk gap is the missing motion model: adding a per-track Kalman filter + OC-SORT's observation-centric heuristics (ORU/OCM/OCR) is a pure-C++, no-new-dependency change that closes most of the gap to SOTA on static cameras and runs at 700+ FPS on CPU. The second gap is that real ReID is half-built: the embedding plumbing exists end-to-end (Det.embedding, EMA-smoothed track embedding, cosine gate, JSON "embedding" field) but nothing produces a discriminative embedding — the comment says detectors pass "a normalized colour histogram," which is far weaker than a learned OSNet/CLIP-ReID descriptor. Wiring a lightweight ONNX ReID head (OSNet x0.25, ~2-4MB, runs on ANE/CUDA you already have) into detect_onnx/tracker would unlock BoT-SORT/Deep-OC-SORT-class identity stability and is the prerequisite for cross-camera work. The third, larger opportunity is MTMC: zm-api owns the DB, ONVIF topology, and ingests per-camera tracks over the stream-socket — it is the natural home for an offline/online cross-camera trajectory-association layer (appearance + camera-topology + time gating), which would be a genuine differentiator vs Frigate/Scrypted, neither of which does cross-camera identity. Honest positioning: we already LEAD Frigate on architecture (Frigate's tracker is a customized Norfair with per-class Kalman tuning and optical-flow CMC, but it explicitly does NOT use ReID — "no models fast enough"), and we have a detect/ReID/ANE stack they lack; but our actual association algorithm currently TRAILS Frigate (they have Kalman + CMC; we have neither). Closing the Kalman/OC-SORT gap flips that. Two production C++ libraries (mikel-brostrom/boxmot native C++ BoT-SORT, and Geekgineer/motcpp with 10 SOTA trackers incl. UCMCTrack/BoostTrack/HybridSORT + ONNX ReID backend) mean we can adopt SOTA association without reimplementing from scratch — but our plugin's clean dependency-free core is worth preserving, so a port-the-algorithm rather than vendor-the-library approach is recommended for the core, with boxmot/motcpp as reference implementations and for ReID ONNX export.

### Video Anomaly Detection (VAD) SOTA — weakly-supervised MIL, training-free / open-set VLM-LLM, zero-shot, and memory-bank normality — mapped to a zm-next "smart alerts beyond motion+class" detect/analytics plugin

The deployable-without-per-site-training frontier has decisively moved from weakly-supervised MIL (RTFM/MGFN/UR-DMU/VadCLIP, all of which need a labeled UCF-Crime/XD-Violence training corpus and cap at ~88-90% AUC) to training-free VLM/LLM pipelines and open-vocabulary customizable VAD. The single most important, immediately-relevant pattern for zm-next is the CASCADE: a cheap always-on filter (motion/feature deviation) that wakes an expensive VLM reasoner only on candidate segments. zm-next ALREADY implements the bottom of that cascade better than almost any competitor (HW-decode → cuda motion gate → YOLO detect → tracker → analytics_rules → llm_event_review with a pluggable IVisionProvider). So the build is NOT a new heavy model; it is a new "anomaly score" layer that turns the existing detect+track+VLM signals into a per-scene normality judgement WITHOUT per-site training. Three deployable, training-free recipes stack onto our existing plugins: (1) Cerberus/SlowFastVAD-style fast-filter→VLM cascade — we already have the fast path; we need rule-deviation scoring + motion-mask prompting on the slow path (llm_event_review). (2) AnyAnomaly-style open-vocabulary customizable VAD — user types "person climbing fence"/"loitering vehicle" as the anomaly, answered by context-aware VQA with NO fine-tune; maps directly onto detect_openvocab + describe_vlm. (3) A memory-bank / CLIP-kNN normality scorer (PatchCore/AnomalyDINO/AF-CLIP lineage) — a fully-local, sub-millisecond, training-free per-camera "have I seen this before?" score that needs only a few hours of that camera's normal footage, no labels. AnomalyRuler/Holmes-VAU contribute the few-normal-shot RULE INDUCTION idea: at install, auto-derive this scene's normal-behavior rules from unlabeled normal clips and feed them to analytics_rules/llm_event_review prompts. Honest read: on object-description + semantic search we are at parity with Frigate 0.15 GenAI and ahead of Scrypted; on true scene-level anomaly scoring (the "unusual-not-just-present" alert) NO open-source NVR ships this yet — it is a genuine differentiator and the recommended build. Skip the MIL trainers (need labels/training, brittle cross-domain). Pilot the CLIP-kNN normality scorer + open-vocab VAD now; watch the heavyweight MLLM-VAU research (Holmes-VAU, CueBench) as a quality ceiling to track, not deploy.

### Natural-language video/event search & multimodal retrieval (CLIP/SigLIP/JinaCLIP embeddings, sqlite-vec/pgvector vector search, temporal grounding, video RAG, agentic query routers) mapped to zm-api event search + the sqlite-vec plan + dashboard/mobile UX.

zm-next's documented NL-search design (docs/Research_Motion_and_LLM_Review.md Phase 3) is genuinely near-SOTA on the hardest, most-overlooked parts: it routes counting/temporal queries to SQL instead of the LLM, resolves relative dates to epoch bounds in code, uses sqlite-vec for in-DB pre-filtering (avoiding pgvector/FAISS post-filter recall loss), and enforces strict-citation grounding with event_id existence-verify. Those are exactly the lessons the 2024-2026 agentic-RAG/Text2SQL literature keeps re-learning, and they are things Frigate does NOT do (Frigate has pure visual+caption CLIP similarity with no SQL router, no counting, no grounded answers). The single biggest SOTA gap vs Frigate is that zm-next plans to embed ONLY VLM-generated text descriptions, while Frigate also embeds the actual thumbnail pixels with a multimodal CLIP model (Jina v1/v2) and searches text->image directly. A caption can drop salient visual attributes ("red car", "blue shirt", text on clothing) that a direct image embedding preserves; the best-known result is to do BOTH and fuse. Two concrete, low-effort upgrades stand out: (1) add a multimodal image embedding (SigLIP 2 or JinaCLIP v2) over the event thumbnail alongside the description embedding, enabling text->image and image->image "find more like this" search with no VLM caption dependency; (2) add a tiny cross-encoder reranker (bge-reranker-v2-m3 / Qwen3-Reranker-0.6B) over the hybrid candidate set, the single highest-ROI RAG quality lever in the recent benchmarks (+17% Recall@5 over hybrid-RRF alone). The current text-embedding model choice (bge-small/nomic/Qwen3-Embedding-0.6B) is fine but EmbeddingGemma-300M and Qwen3-Embedding-0.6B are the 2025 short-text/CPU sweet spot with Matryoshka truncation to keep vectors small. Temporal grounding and heavy video-RAG (VideoRAG, Video-RAG, TRACE) are research-grade and overkill here -- zm-next already has the event/track structure that those papers try to recover from raw video, so it should stay event-centric and watch, not adopt. Where Frigate clearly leads today: shipped, polished local semantic search UX in the dashboard with visual similarity and "triggers." Where zm-next leads: grounded NL Q&A with SQL routing, counting, and verifiable deep-links -- a real differentiator if shipped.

The tracker even has optional ReID/appearance-embedding hooks wired (cosine sim + EMA), just lacking an embedding producer. The biggest concrete, low-effort wins are: (1) make INT8 the default export/calibration path for detect_onnx — YOLO26's NMS-free, DFL-free design holds mAP under INT8 where YOLOv11/12 drop 3-7%, giving 1.5-3.3x on CPU/OpenVINO/NPU and unlocking Tensor-Core INT8 + Coral-class accelerators for cheap many-camera scale; (2) wire a lightweight ReID embedding head (LITE-style in-detector features, or a tiny OSNet) into the already-present tracker hooks; (3) adopt the ORT TensorRT EP (not raw TensorRT) on Linux/CUDA to get fp16/INT8 kernel fusion through the existing one-codebase abstraction. Higher-value bets to pilot: Apple FastVLM (0.5B/1.5B, FastViT-HD encoder, 85x faster TTFT) as the describe_vlm backend on the ANE/MLX, replacing the generic Moondream/Qwen path for real-time-ish scene description; and an explicit per-frame/temporal early-exit gate (E4/FrameExit/NoScope-style) layered on motion_gate to skip redundant frames. Where competitors lead: nothing fundamental — Frigate has a more mature off-the-shelf model zoo and Coral integration, and DeepStream has hardware-batched NVDEC→inference fusion we approximate in software, but our cross-platform ORT abstraction is the better long-term bet.

### Action recognition, pose-based event detection, and audio event detection SOTA mapped to zm-next detect_pose, analytics_rules, and audio_detect

zm-next is well-positioned architecturally but under-uses learned temporal models. The biggest gap: detect_pose emits high-quality per-frame COCO-17 skeletons, but NOTHING consumes them temporally — analytics_rules ignores pose entirely and works purely on tracker bbox geometry (intrusion/linecross/loiter as hand-coded math). So fall/fight/collapse/loiter-with-intent are not actually recognized as actions today; loiter is just a dwell timer. The single highest-leverage move is a new lightweight skeleton-sequence action classifier stage that consumes detect_pose output and emits action events into analytics_rules. Two mature, ONNX-exportable families exist: (1) GCN-based ST-GCN++/CTR-GCN/PoseConv3D (PYSKL/MMAction2) — top accuracy, heavier; (2) a YOLO-pose-keypoints -> small LSTM/TCN/Transformer sequence head — the dominant practical 2024-2025 surveillance pattern (97-98% fall accuracy, ~1% false alarm, real-time on CPU). Recommendation: pilot the lightweight TCN/Transformer-on-keypoints head first (it slots directly onto the existing 17-keypoint stream and your ANE path), keep PoseConv3D/STGCN++ as a watch for the multi-person/occlusion-heavy case. For audio_detect, the architecture (FFmpeg decode -> resample 16k mono -> sliding window -> ONNX -> top-k) is already SOTA-shaped, but a YAMNet backbone is ~0.37 mAP vs SOTA ~0.49 mAP. Drop-in swapping the ONNX model to CED (mispeech/ced-base, ViT, 49.0 mAP, HF + custom code, ~10M params) or EfficientAT (MobileNetV3 distilled from transformers, 0.483 mAP, scales to CPU-tiny) is near-zero-effort, high-impact and needs no plugin code change beyond labels/window length. CLAP (LAION-CLAP, 89.5% ESC-50 zero-shot) enables open-vocabulary audio events ("glass break","scream","gunshot") via text prompts with no retraining — a strong pilot that mirrors your detect_openvocab philosophy. Audio-visual fusion (XD-Violence, MACIL-SD, HyperVD) for fight/violence detection is real SOTA but heavy and is a watch given your event-bus architecture could fuse audio_event + pose-action cheaply at the analytics layer instead. Competitor reality check: Frigate has YAMNet audio (same tier you already match) and NO fall/action recognition; nobody in the prosumer NVR space ships skeleton-action — adding it would put zm-next ahead. Finally, your documented describe_vlm/llm_event_review direction overlaps with the strongest research trend (Video-LLM temporal reasoning, Qwen2.5-VL/TEMPURA) — but those models still struggle with fine-grained temporal action discrimination, so use them for event-level review/explanation, not as the primary fall/fight detector.

### Competitive AI NVR ecosystem, streaming transport SOTA, and privacy/redaction for zm-next

zm-next's AI tier is already at or ahead of the open-source NVR field on the model layer: detect_onnx/seg/openvocab/pose, tracker with ReID, recognize_face, lpr, audio_detect, and describe_vlm (mlx-vlm Qwen3-VL on ANE) collectively match or exceed Frigate 0.17, Scrypted NVR, and Viseron. On PRIVACY, zm-next has a structural ADVANTAGE to press: it is fully local/on-prem (no cloud GenAI by default), and the review_export tube-synopsis feature is the natural home for on-device face/plate redaction. EgoBlur (Meta, Apache-2.0) and DeepPrivacy2 are drop-in SOTA for redaction; the EU AI Act (prohibited-practices live Feb 2025, transparency obligations Aug 2026) plus GDPR DSAR redaction are concrete product-positioning levers vs cloud-dependent competitors. Net: zm-next leads on local-first models and codec breadth; it trails Frigate on shipped search/GenAI-review UX and should prioritize sqlite-vec NL search + redaction-on-export to leapfrog into a "private, local, searchable" position.

**Findings:**

- [**adopt-now** · high/high] **Semantic / natural-language event search (CLIP-style embeddings)** → `NEW llm_event_review plugin + sqlite-vec store; zm-api (search API/serve); describe_vlm (caption source); detect_onnx (crop source); zm-dashboard / zm-mobile / zmNinjaNg (query UI)`
  - SOTA: Frigate 0.15-0.17 Semantic Search uses CLIP embeddings over tracked-object thumbnails for text + visual-similarity search, plus 'Search Triggers' that auto-act when an object matches a text/image. SOTA embedders: SigLIP / SigLIP2 (sigmoid loss, better recall than CLIP), jina-clip-v2 (89 languages, 512x512 input, Dec 2024), jina-embeddings-v4 (Qwen2.5-VL-3B backbone, Jun 2025). Local vector store: sqlite-vec (handles tens of thousands to low-millions of vectors on one machine, no extra service), or pgvector inside the existing zm-api Postgres-class DB.
  - Relevance: This is the single largest feature gap vs Frigate. zm-next already produces all the inputs (detect crops, describe_vlm captions, tracker tubes) but has no embedding index or NL query path. The documented sqlite-vec direction is correct; describe_vlm captions can be embedded as text and object crops as images for hybrid search. zm-api owns the DB and is the natural query surface for dashboard/mobile.
- [**adopt-now** · medium/high] **GenAI object descriptions + review summaries with risk classification** → `describe_vlm (add structured JSON output: title/description/risk_class); zm-api (review item ingest/serve); output_mqtt / output_webhook (notification payload); zm-dashboard review UI`
  - SOTA: Frigate 0.16/0.17 GenAI: per-object natural-language descriptions (feeds semantic search) AND structured review-item summaries (LLM emits title + description + activity class dangerous/suspicious/normal) shown automatically in the review UI. Providers: Ollama (local), OpenAI, Gemini. Home Assistant 'LLM Vision' integration popularized 'a delivery driver in a brown uniform left a box' style notifications over MQTT.
  - Relevance: zm-next's describe_vlm (local mlx-vlm Qwen3-VL on ANE) is architecturally AHEAD of Frigate here — Frigate mostly leans on external/Ollama providers, zm-next runs a real local VLM on the Neural Engine. The gap is purely product: structured output (title/desc/risk-class) and surfacing it in a review/timeline UI, not raw capability. This is a low-effort, high-visibility win that directly differentiates 'fully local GenAI'.
- [**adopt-now** · medium/high] **Review / timeline UX as the primary interaction model** → `review_export (tube synopsis); zm-api (motion synopsis optimiser+render+serve, already in progress); zm-dashboard / zm-mobile timeline + synopsis player`
  - SOTA: Frigate 0.17 'Review' replaces raw-event scrolling: review items on a vertical timeline, bandwidth-optimized low-fps/low-res preview grid, GenAI summaries inline, severity coloring. Milestone acquired BriefCam (video-synopsis pioneer) in 2024 — synopsis/review is now a commercial differentiator. Academic tube-set synopsis (ScienceDirect 2024, YOLOv4+Kalman tubes) underpins zm-next's review_export approach.
  - Relevance: zm-next's NEW review_export (object tubes + plates synopsis) is conceptually AHEAD of Frigate's preview-grid review (true synopsis vs sampled previews) and aligns with the BriefCam/Milestone direction. But review UX lives in apps; the synopsis must be served and browsable. This is where zm-next can claim 'BriefCam-style synopsis, fully local, free'.
- [pilot · medium/medium] **Detector abstraction across heterogeneous accelerators** → `detect_onnx (add execution-provider backends: OpenVINO, TensorRT, Hailo, RKNN, Coral/EdgeTPU); HwBackend abstraction; decode_ffmpeg`
  - SOTA: Frigate builtin detector types: cpu, edgetpu, hailo8l, onnx, openvino, rknn, tensorrt; multiple detectors in dedicated processes; 0.17 adds CUDA Graphs, OpenVINO NPU, RKNN auto-ONNX->RKNN conversion, Apple Silicon NPU via IPC, community MemryX/Degirum/Synaptics. Constraint: cannot mix detector backends for one object-detection task. Scrypted: CoreML(ANE)/OpenVINO/ONNX/Coral plugins.
  - Relevance: zm-next's C-ABI plugin model (dlopen, per-thread StageRunner, HwBackend with VideoToolbox/CUDA/Metal/ANE validated bit-exact) is a CLEANER abstraction than Frigate's process-per-detector config — and zm-next already runs co-located Apple ANE + Linux CUDA. zm-next is at/above SOTA here; the gap is breadth of shipped backends (no Hailo/Coral/RKNN/OpenVINO-NPU yet). Hailo-8 and Coral are the cheap-edge backends competitors lean on.
- [**adopt-now** · low/medium] **SOTA detection models (YOLOv11/v12, RT-DETR/RF-DETR)** → `detect_onnx (model weights/export); bench/metal harness (validation); detect_seg / detect_pose (newer seg/pose variants)`
  - SOTA: YOLOv11 (Oct 2024, ~22% fewer params than v8m at higher mAP), YOLOv12 (Feb 2025, attention-centric: Area-Attention A2 + R-ELAN + FlashAttention, beats RT-DETR at larger scales in speed/params), RF-DETR (Roboflow, transformer SOTA on COCO), YOLOv9-on-Coral quantized (Frigate 0.17 default upgrade over mobiledet). RT-DETR removes NMS for cleaner end-to-end latency.
  - Relevance: zm-next's detect_onnx is model-agnostic (ONNX), so adopting YOLOv11/v12/RF-DETR is a model-swap, not an engineering project. Frigate just moved its Coral default to YOLOv9; staying on older YOLO would be a quiet accuracy deficit. Low effort to validate newer weights through the existing ONNX path + the bit-exact Metal/ANE bench harness.
- [**adopt-now** · medium/high] **Sub-second WebRTC live (WHEP/WHIP) + go2rtc-class gateway** → `output_webrtc (WHEP/WHIP endpoint); zm-api live streaming (signaling); signaling/ Node bridge (replace with WHEP); zm-dashboard/zm-mobile players`
  - SOTA: go2rtc delivers ~0.5s WebRTC latency browser-direct, universal RTSP/RTMP/HLS/MSE/WebRTC restream, WHEP/WISH support; it is the de-facto live layer behind Frigate and Home Assistant. WHEP (WebRTC-HTTP Egress) is the standardized one-shot WebRTC playback signaling; WHIP for ingest.
  - Relevance: zm-api already owns live streaming (WebRTC/HLS/MSE) and zm-next has output_webrtc (LibDataChannel) + output_mse. The gap vs go2rtc is (a) standardized WHEP/WHIP signaling instead of the custom Node bridge in signaling/, and (b) a single gateway that restreams any input. Adopting WHEP makes zm-api interoperable with standard players and removes bespoke signaling. This is near-SOTA already; WHEP is the polish.
- [pilot · medium/medium] **H.265/HEVC over WebRTC (avoid transcode)** → `output_webrtc (H.265 payload); decode_ffmpeg / capture_rtsp_multi (passthrough path); zm-api streaming; zm-mobile (iOS HEVC)`
  - SOTA: Chrome ships RTC H.265 decode from v127 (Intent-to-Ship landed); HEVC decode in WebCodecs; HW HEVC decode coverage ~99% macOS, ~90% iOS, ~86% Android, ~75% Windows; draft-ietf-avtcore-hevc-webrtc IETF profile. This lets servers stream camera-native H.265 without transcoding to H.264.
  - Relevance: Most modern IP cameras emit H.265; zm-next currently leans H.264 for WebRTC (output_webrtc) which forces transcode or restricts cameras. With Chrome/Safari/iOS HEVC-over-WebRTC now viable, zm-api could passthrough camera H.265 to browsers, cutting CPU/GPU transcode load — a direct efficiency win on the M4 Pro / CUDA boxes. Apple ecosystem (Safari/iOS) is HEVC-strong, aligning with zm-mobile.
- [watch · high/medium] **AV1 + SVC for bandwidth-efficient adaptive WebRTC** → `output_webrtc (AV1+SVC layering); zm-api ABR/streaming; zm-mobile (cellular viewing)`
  - SOTA: AV1 is the first codec with SVC by default; AV1 RTP payload + W3C webrtc-svc let one encoder emit layered streams (temporal/spatial) adapting per-viewer without server transcode. Chrome/Edge ship AV1; Safari AV1 still limited. AI encoders (Visionular Aurora1) push AV1 bitrate savings at low latency.
  - Relevance: Useful for remote/mobile viewing of many cameras over constrained links (zm-mobile on cellular). SVC single-encode-many-layers fits zm-next's per-stream model and avoids transcode fan-out. But AV1 encode is CPU-heavy without HW encoders, and Safari/iOS support lags — so impact is real but bounded near-term. Watch HW AV1 encode on M4 Pro / NVIDIA.
- [watch · high/low] **Media over QUIC (MoQ) next-gen transport** → `zm-api streaming transport (future); output_webrtc (future MoQ sibling)`
  - SOTA: IETF MOQ WG; draft-ietf-moq-transport (pre-RFC, draft ~05+ Oct 2024) over QUIC/WebTransport; sub-second demos. 2025 milestones: Cloudflare launched first MoQ relay network (330+ cities); nanocosmos first production MoQ at IBC 2025. Still experimental, not widely deployed, browser support immature.
  - Relevance: Strategically interesting (unifies live + scalable fan-out + low latency) but premature for a self-hosted NVR: no stable spec, thin browser support, and zm-next's value is local LAN where WebRTC already wins. Track it so zm-api's transport layer stays MoQ-ready, but do not build now.
- [**adopt-now** · medium/high] **On-device face & license-plate redaction (export + live)** → `NEW privacy_mask plugin (EgoBlur/LPDNet detectors via detect_onnx); review_export (redacted synopsis/export); store (redacted clips); lpr / recognize_face (reuse detections)`
  - SOTA: EgoBlur (Meta, Apache-2.0): two FasterRCNN/ResNeXt detectors (face, plate), ~104M params each, trained 23M images/790M boxes; CLI blurs images+video; Gen2 (2025) improved. DeepPrivacy2 (WACV 2023): style-GAN realistic full-body + face anonymization with text-guided edits (preserves utility vs blur). NullFace (2025): training-free localized diffusion face anonymization. NVIDIA LPDNet for plate detect. EU/GDPR vendors (Facit, Brighter AI/blurit, Sighthound Redactor) sell this as a paid feature.
  - Relevance: review_export already isolates object tubes + plate crops — redaction is a natural bolt-on. EgoBlur is the pragmatic adopt: Apache-2.0, ONNX-exportable through detect_onnx, runs on the existing HW backends. This unlocks GDPR DSAR-compliant exports and 'blur bystanders' that Frigate notably LACKS (Frigate masks don't redact). A genuine feature LEAD opportunity, not catch-up.
- [pilot · low/high] **Regulatory positioning: EU AI Act + GDPR as product levers** → `Product positioning + docs; recognize_face / tracker ReID (consent gating, audit log); privacy_mask (DSAR redaction); zm-api (audit/retention policy)`
  - SOTA: EU AI Act: prohibited practices live Feb 2025 (incl. untargeted scraping of CCTV to build face-rec DBs; emotion recognition in workplaces/education; biometric categorization); real-time remote biometric ID in public spaces largely banned for law enforcement w/ narrow exceptions + FRIA + EU DB registration; transparency obligations for biometric/emotion systems applicable 2 Aug 2026. GDPR treats CCTV footage as PII; DSAR responses require redacting third parties.
  - Relevance: zm-next is local/on-prem with optional, off-by-default ReID and local-only GenAI — structurally well-placed for compliance vs cloud-GenAI competitors. Concrete moves: (1) keep recognize_face/ReID opt-in with clear consent docs; (2) ship redaction-on-export for DSAR; (3) add transparency/audit logging for biometric features; (4) avoid emotion-recognition and untargeted-DB framings. Position 'private by design, EU-AI-Act-aware' as differentiation vs Frigate/Scrypted/cloud.
- [pilot · medium/medium] **Audio transcription / speech events** → `audio_detect (add STT via faster-whisper/sherpa-onnx); llm_event_review/sqlite-vec (index transcripts); zm-api search`
  - SOTA: Frigate 0.17 adds local audio transcription via sherpa-onnx or faster-whisper, with live speech event analysis (searchable speech-to-text alongside object events).
  - Relevance: zm-next has audio_detect (sound classification) but not speech-to-text. faster-whisper/sherpa-onnx are local, ONNX-friendly, and would feed the same semantic-search index (transcripts are highly searchable). Modest effort given the audio path already exists; complements NL search.
